# ChatGPT Exporter — File Extraction Capability Prompt Pack

## Purpose

Use this prompt pack to modify the forked `danveitch76/chatgpt-exporter` repository in controlled chunks. The goal is to add a dedicated file extraction mode for both:

1. Files uploaded by the user into ChatGPT conversations.
2. Files generated by ChatGPT and presented as downloadable artefacts.

The implementation should preserve the existing exporter behaviour and add a new export mode rather than replacing existing conversation export functions.

## Repository Context

The repository is a Tampermonkey userscript project using TypeScript, Preact, Vite and JSZip.

Relevant existing files:

```text
src/api.ts
src/constants.ts
src/main.tsx
src/ui/ExportDialog.tsx
src/ui/Menu.tsx
src/ui/MenuItem.tsx
src/exporter/json.ts
src/exporter/markdown.ts
src/exporter/html.ts
src/utils/download.ts
src/utils/dom.ts
```

Existing useful capabilities:

```text
fetchAllConversations()
fetchConversation()
fileDownloadApi(id)
fetchImageFromPointer(uri)
replaceImageAssets(conversation)
JSZip dependency already available
```

## Engineering Principles

Do not attempt a large one-shot rewrite.

Implement this in chunks:

1. Types and data model.
2. Conversation scanning.
3. Download resolution.
4. Inventory generation.
5. ZIP packaging.
6. User interface wiring.
7. Validation and build.

Each chunk must compile before moving to the next.

Do not remove existing export modes.

Do not weaken current conversation export functionality.

Do not assume every file reference is downloadable. Expired or metadata-only artefacts must still appear in the inventory with a clear failure reason.

# Chunk 2 — Add Defensive Conversation Scanner

```text
Add a scanner that finds likely uploaded and generated file references in a ChatGPT conversation object.

Create:

src/file-extractor/scanConversation.ts

Export:

scanConversationForFiles(conversation: ApiConversationWithId): FileReference[]

The scanner must:

1. Iterate over `conversation.mapping`.
2. Ignore empty nodes safely.
3. Inspect each message content and metadata recursively.
4. Detect file references in common shapes:

- file_id
- fileId
- id values that look like file identifiers
- asset_pointer
- audio_asset_pointer
- image_asset_pointer
- sandbox:/mnt/data/...
- /mnt/data/...
- sediment://...
- download URLs or paths containing `/files/download/`
- filenames in annotations or attachments

5. Classify source type using conservative rules:

- `image` for sediment/image/audio/video asset pointers and multimodal image assets.
- `generated` for sandbox paths, `/mnt/data/`, code execution outputs, file citations or assistant artefacts.
- `uploaded` for user message attachments or file metadata found on user-authored messages.
- `unknown` when uncertain.

6. Extract filename and extension where possible.
7. Preserve raw object/value location through `sourceField`.
8. Never throw if fields are missing or unexpected.

Add helper functions inside the module if needed:

- walkObject()
- inferFilename()
- inferExtension()
- inferSourceType()
- normaliseAssetPointer()

Acceptance criteria:

- TypeScript compiles.
- Scanner returns an empty array rather than throwing on unexpected objects.
- Existing exporter modes are unaffected.
```

# Chunk 3 — Add File Download Resolver

```text
Add download resolution for extracted file references.

Create:

src/file-extractor/downloadManager.ts

Export:

resolveAndDownloadFile(ref: FileReference): Promise<{ blob?: Blob; status: FileDownloadStatus; failureReason?: string; resolvedFilename?: string; mimeType?: string; sizeBytes?: number }>

Implementation requirements:

1. Reuse existing ChatGPT backend route pattern from `api.ts` for `/files/download/:id`.
2. If `ref.fileId` exists, call the backend download endpoint and then fetch the returned `download_url`.
3. If `ref.assetPointer` starts with `sediment://`, strip the prefix and resolve it through the same file download route.
4. If only `sandbox:/mnt/data/...` or `/mnt/data/...` exists, do not pretend it is downloadable unless a file identifier is also present. Mark as `metadata_only`.
5. If a direct HTTP URL is available and same-origin/authenticated fetch succeeds, download it. Otherwise fail safely.
6. Return structured status and failure reason. Do not throw unless the caller passed invalid input.
7. Include timeout handling or clear fetch failure handling.
8. Preserve content type and blob size when available.

If `fileDownloadApi` is not exported from `api.ts`, refactor `api.ts` minimally to export a safe helper:

export async function fetchFileDownloadUrl(fileId: string): Promise<string>

Do not expose access tokens in logs.

Acceptance criteria:

- TypeScript compiles.
- Download failures are captured as inventory failures.
- No existing export behaviour changes.
```

# Implementation Risk Register

| Risk | Impact | Mitigation |
|---|---:|---|
| ChatGPT internal object shapes change | High | Use recursive, defensive scanning and preserve unknown references. |
| Older generated files expired | High | Inventory must record metadata-only and failed downloads. |
| File identifiers hidden in multiple locations | High | Scan content, metadata, annotations, multimodal parts and text. |
| Large accounts hit rate limits | Medium | Add progress, batching and failure capture. |
| Duplicate file references inflate ZIP | Medium | Deduplicate but preserve duplicate rows. |
| UI locks during long export | Medium | Add progress and finalisation handling. |
| Browser memory pressure | Medium | Keep first version simple; later add chunked ZIPs if needed. |
